MINI MÓDULO “FALTANTES” (SEÑAL HUMANA) + ALERTA OPERATIVA + AUDITORÍA CONTRA INVENTARIO (ANTI-ROBO/MERMA)

Objetivo (CERRADO)

Crear un mini-módulo Faltantes para que cualquier empleado pueda reportar de inmediato que algo “se acabó”, y que eso:
	1.	Genere una alerta operativa para reponer/comprar.
	2.	Genere una alerta de auditoría si el sistema de inventario indicaba que había stock (para detectar robo/merma/error de control).

Este módulo NO reemplaza inventario. Es una señal humana + trazabilidad + auditoría.

⸻

Reglas “en piedra”
	1.	Cualquier empleado puede: Reportar faltante.
	2.	Reportar faltante NO ajusta el stock automáticamente.
	3.	Si el faltante es de INSUMO (inventario):
	•	Guardar snapshot del stock del sistema en el momento del reporte.
	•	Si el sistema mostraba on_hand_qty_base > 0 (con tolerancia), crear alerta de auditoría para gerente.
	4.	Si el faltante es de PRODUCTO DEL MENÚ:
	•	Se reporta igual (alerta operativa), pero NO desactiva el producto automáticamente.
	•	Solo roles autorizados pueden marcar el producto como “NO DISPONIBLE” (opcional).
	5.	Debe existir un panel exclusivo de gerente: Auditoría de Faltantes (discrepancias).

⸻

Alcance (CERRADO)
	•	Pantallas:
	1.	Reportar faltante (todos, mobile-first)
	2.	Faltantes activos (lectura todos; acciones según permisos)
	3.	Auditoría de faltantes (solo gerente)
	•	Datos:
	•	Tablas nuevas: inv_shortages, inv_shortage_events, inv_audit_alerts
	•	Integración:

	•	Notificaciones:
	•	Banner/popup interno (pantallas staff) + sonido opcional (solo interno, no cliente QR)

NO implementar órdenes de compra automáticas aquí (solo sugerencia de compra).

Lógica de negocio (CERRADA)

A) Reportar faltante (INV_ITEM)

Endpoint: POST /api/shortages/report
Input:
	•	entity_type=INV_ITEM
	•	inv_item_id
	•	notes opcional

Proceso:
	1.	Buscar si ya existe shortage con status OPEN o ACKNOWLEDGED para ese inv_item_id.
	•	Si existe:
	•	report_count += 1
	•	last_reported_at = now
	•	crear inv_shortage_events REPORTED
	•	retornar shortage existente
	•	Si NO existe:
	•	leer inv_items.on_hand_qty_base y avg_cost_per_base_uom
	•	calcular sugerido compra:
	•	if par_level_qty_base > 0: suggested = max(0, par_level - on_hand)
	•	else if reorder_point_qty_base > 0: suggested = max(0, reorder_point - on_hand)
	•	else suggested = null
	•	crear inv_shortages status=OPEN con:
	•	snapshots: system_on_hand_qty_base_snapshot = on_hand
	•	system_avg_cost_snapshot = avg_cost
	•	suggested_purchase_qty_base = suggested
	•	crear inv_shortage_events REPORTED
	2.	Regla de auditoría (anti-robo/merma):
	•	tolerancia: on_hand > 0.01 (hardcode MVP)
	•	si snapshot system_on_hand_qty_base_snapshot > 0.01:
	•	set shortage:
	•	audit_flag=true
	•	audit_reason=REPORTED_ZERO_BUT_SYSTEM_POSITIVE
	•	audit_status=OPEN
	•	crear inv_audit_alerts:
	•	alert_type=SHORTAGE_DISCREPANCY
	•	severity=HIGH
	•	inv_item_id y shortage_id
	•	message: “Faltante reportado pero el sistema indica stock positivo. Revisar posible merma/robo/receta/unidad.”
	3.	Anti-spam extra:
	•	si report_count >= 3 el mismo business_date:
	•	audit_reason=MULTIPLE_REPORTS_SAME_DAY
	•	crear alerta MEDIUM si no existe ya una abierta para ese shortage.

IMPORTANTE: NO ajustar stock por este reporte humano.

⸻

B) Reportar faltante (MENU_PRODUCT)

Input:
	•	entity_type=MENU_PRODUCT
	•	menu_product_id
	•	notes opcional

Proceso:
	•	Igual: crear shortage OPEN si no existe; si existe, sumar contador.
	•	NO modificar el producto automáticamente.
	•	Si usuario tiene MENU_TOGGLE_AVAILABILITY:
	•	permitir acción separada: “Marcar NO DISPONIBLE” que setea product.active=false o available=false según tu modelo actual y audita.

⸻

C) Gestión de estados del faltante

Acciones:
	•	ACKNOWLEDGE (cocina/gerente): PATCH /api/shortages/:id/ack
	•	RESOLVE (cocina/gerente): PATCH /api/shortages/:id/resolve
	•	CLOSE (gerente): PATCH /api/shortages/:id/close

Cada acción:
	•	cambia status
	•	registra inv_shortage_events
	•	requiere nota opcional (CLOSE: nota obligatoria)

⸻

D) Auditoría gerente (inv_audit_alerts)

Endpoints:
	•	Listar OPEN: GET /api/audit-alerts?status=OPEN
	•	ACK: PATCH /api/audit-alerts/:id/ack (nota opcional)
	•	CLOSE: PATCH /api/audit-alerts/:id/close (nota obligatoria)

Al ACK:
	•	opcional: set shortage.audit_status=INVESTIGATING y audit_owner=gerente.
Al CLOSE:
	•	set shortage.audit_status=RESOLVED (si corresponde) y guardar audit_notes.

Definition of Done (global)
	•	Reporte de faltantes disponible para todos.
	•	Shortage OPEN único por item, con contador/eventos.
	•	Discrepancia (stock>0) genera alerta de auditoría para gerente.
	•	Gerente puede ACK/CLOSE con notas obligatorias.
	•	Sin ajustes automáticos de stock por reporte humano.
	•	Alertas internas en tiempo real
