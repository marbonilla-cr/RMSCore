DECISIÓN PRINCIPAL (OBLIGATORIA):
MODE FULL + Recetas:
1) Al confirmar OrderItem a status='SENT', ejecutar primero FULL deduction (receta).
2) SOLO si FULL deduction finaliza OK (stock validado y consumos registrados), entonces ejecutar BASIC deduction (decrementPortions del producto).
3) Si FULL deduction falla por stock insuficiente, BLOQUEAR la confirmación a SENT y NO ejecutar BASIC. Debe devolver error claro al usuario.

Esto elimina la inconsistencia de “additive a ciegas”. FULL manda y BASIC se aplica después como reflejo operativo.

E) IDempotencia y orden de ejecución (B)
Implementar en onOrderItemsConfirmedSent(orderId, orderItemIds[], userId):
1) Por cada orderItemId:
   - Si ya existe inventory_deductions row => saltar (idempotente) y NO tocar BASIC.
2) Si no existe:
   - Resolver recipe lines
   - Validar stock EP/AP suficiente
   - Consumir stock y registrar movements
   - Insertar inventory_deductions con payload consumos
3) SOLO después de confirmar consumos FULL exitosos para ese item:
   - Ejecutar BASIC decrementPortions(productId, qty) exactamente una vez (idempotente) para ese orderItemId
   - Para idempotencia de BASIC en modo FULL:
     - agregar marca en inventory_deductions payload o una columna basicDeductedAt para no repetir BASIC si reintenta.
4) Si FULL falla -> throw error y NO descontar BASIC y NO permitir SENT.
