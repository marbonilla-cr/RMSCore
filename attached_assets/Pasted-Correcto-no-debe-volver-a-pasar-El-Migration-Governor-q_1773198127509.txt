Correcto, no debe volver a pasar. El Migration Governor que implementamos tiene exactamente ese propósito — pero tiene un hueco que hay que cerrar.

**Por qué pasó:**

`backfillMigrationLog` se usó durante la migración inicial de Replit a Railway para "sincronizar" el estado sin correr el SQL. Fue un atajo que dejó registros fantasma. Cuando después se agregó `0010_print_bridges.sql`, ya había un registro fantasma de migraciones anteriores, y el sistema asumió que todo estaba al día.

**Por qué no debe volver a pasar:**

Con `propagateMigrations` activo, el flujo ahora es:
1. Startup → lee archivos en `/migrations/`
2. Compara contra `schema_migrations`
3. Si un archivo no está registrado → ejecuta el SQL primero, luego registra
4. Si ya está registrado → lo salta

El problema es que los registros fantasma de `backfillMigrationLog` siguen en la DB. Si alguna vez se agrega una migración que modifica una tabla que "supuestamente" ya fue creada, va a fallar silenciosamente de nuevo.

## Prompt para Replit Agent

Para garantizar que el bug de registros fantasma no vuelva a ocurrir, necesito una verificación de integridad en el Migration Governor.

**En `server/provision/migrate-tenants.ts`**, agregar una función `verifyMigrations` que corra después de `propagateMigrations` en `syncAllTenantsAtStartup`:

```typescript
async function verifyMigrations(schemaName: string, client: PoolClient): Promise<void> {
  const migrationsDir = path.join(process.cwd(), 'migrations');
  const files = fs.readdirSync(migrationsDir)
    .filter(f => f.endsWith('.sql'))
    .sort();

  for (const filename of files) {
    // Verificar que cada migración registrada realmente produjo su efecto
    // Por ahora: solo loguear si hay discrepancia entre registradas y archivos
    const { rows } = await client.query(
      `SELECT 1 FROM public.schema_migrations 
       WHERE schema_name = $1 AND filename = $2`,
      [schemaName, filename]
    );
    
    if (!rows.length) {
      console.warn(`[migrate-verify] ⚠ ${schemaName}: ${filename} NO está registrada — debería haberse aplicado`);
    }
  }
  console.log(`[migrate-verify] ✓ ${schemaName}: verificación completa`);
}
```

Y en `syncAllTenantsAtStartup`, después de cada `propagateMigrations`:

```typescript
await propagateMigrations('public');
await verifyMigrations('public', client); // ← agregar

// y para cada tenant activo:
await propagateMigrations(tenant.schema_name);
await verifyMigrations(tenant.schema_name, client); // ← agregar
```

Esto garantiza que en cada startup los logs muestren claramente si alguna migración quedó sin aplicar, en lugar de fallar silenciosamente.
