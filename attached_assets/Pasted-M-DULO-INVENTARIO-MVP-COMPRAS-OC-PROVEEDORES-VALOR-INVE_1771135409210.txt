MÓDULO INVENTARIO (MVP) + COMPRAS + OC + PROVEEDORES + VALOR INVENTARIO + RECETAS (BOM) + CONSUMO AUTOMÁTICO AL ENVIAR A COCINA

Contexto y decisiones ya tomadas (NO cambiar)
	•	Rebajar inventario al enviar a cocina (Opción A).
	•	Costo de inventario: promedio ponderado (Weighted Average Cost).
	•	UoM: se compra en unidad grande (kg/l/caja) y se consume en unidad pequeña (g/ml/unidad) mediante conversiones.
	•	Debe existir switch activo/inactivo para que un producto del menú se controle bajo inventario o no (inventario opcional por producto).
	•	El módulo debe trabajar contra los pedidos ya existentes (orders/order_items) y sumar un nuevo tipo: insumos/items (inventario).

Alcance (cerrado)

Implementar un módulo de inventario simple y usable con:
	1.	Catálogo de Insumos (inventory items) con unidades base y conversiones.
	2.	Proveedores.
	3.	Órdenes de compra (OC) con recepción parcial y generación de movimientos.
	4.	Kardex/Movimientos (única forma de afectar stock; NO editar stock directo salvo ajuste controlado).
	5.	Conteo físico con ajuste auditable.
	6.	Valor inventario con promedio ponderado.
	7.	Recetas/BOM por producto del menú, para consumo automático al enviar a cocina.
	8.	Integración: al enviar items a cocina -> generar consumo de insumos según BOM, solo si el producto está marcado como “controla inventario”.
	9.	Reversos: si un ítem se anula/void y ya había ido a cocina -> devolver consumo al stock (movimiento reverso).

NO implementar lotes/vencimientos ni multi-bodega en esta entrega.

Integraciones con pedidos existentes (NO refactor, solo hooks)

Evento 1: “Enviar a cocina”

Cuando un order_item pasa a estado enviado a cocina (o se crea el ticket KDS), ejecutar:

SI menu_product.inventory_control_enabled=true Y existe receta activa:
	•	Para cada inv_recipe_line:
	•	consume_qty_base = qty_base_per_menu_unit * order_item.qty
	•	Aplicar waste opcional: consume_qty_base *= (1 + waste_pct)
	•	Insertar inv_movements:
	•	type = CONSUMPTION
	•	qty_delta_base = -consume_qty_base
	•	reference_type = ORDER_ITEM
	•	reference_id = order_item.id
	•	created_by_employee_id = usuario que envía a cocina
	•	Actualizar stock y costo promedio (ver algoritmo más abajo)

Si no hay receta o inventory_control_enabled=false: no hacer nada.

Evento 2: “Anulación/void de ítem ya enviado a cocina”

Si un order_item se anula y ya generó consumo:
	•	Insertar movimientos REVERSAL por cada línea consumida:
	•	qty_delta_base = +consume_qty_base (mismo cálculo que el consumo original)
	•	reference_type = ORDER_ITEM
	•	reference_id = order_item.id
	•	nota: “Reversal por anulación”
	•	No recalcular costo promedio por reverso (mantener avg_cost actual) o usar el avg_cost actual para value_delta. (MVP: usar avg_cost actual, auditable.)

Importante: debe haber idempotencia para no reversar dos veces. Usar una marca:

	•	O tabla inv_order_item_consumptions (recommended)
o
	•	verificar si ya existe movimiento type CONSUMPTION para ese order_item y si ya existe REVERSAL.

Tabla recomendada inv_order_item_consumptions (para trazabilidad/idempotencia)
	•	id
	•	order_item_id
	•	recipe_id
	•	status ENUM CONSUMED|REVERSED
	•	created_at, reversed_at nullable

⸻

Costeo: promedio ponderado (WAC) — regla “en piedra”

Aplicar solo en movimientos RECEIPT (compra recibida). Para cada RECEIPT:
	•	old_qty = inv_items.on_hand_qty_base
	•	old_avg = inv_items.avg_cost_per_base_uom
	•	receipt_qty = qty_delta_base (positivo)
	•	receipt_unit_cost = unit_cost_per_base_uom (obligatorio)
	•	new_qty = old_qty + receipt_qty
	•	new_avg = if new_qty==0 then 0 else ((old_qty*old_avg) + (receipt_qty*receipt_unit_cost)) / new_qty

Actualizar:
	•	inv_items.on_hand_qty_base = new_qty
	•	inv_items.avg_cost_per_base_uom = new_avg
	•	inv_items.last_cost_per_base_uom = receipt_unit_cost

Para CONSUMPTION y ADJUSTMENT:
	•	Solo cambia on_hand_qty_base sumando qty_delta_base
	•	avg_cost_per_base_uom no cambia (MVP)
	•	value_delta usar avg_cost_per_base_uom actual para registrar costo teórico
