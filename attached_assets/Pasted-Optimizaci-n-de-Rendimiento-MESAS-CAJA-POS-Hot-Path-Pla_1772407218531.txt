T003: POST /api/pos/pay — BATCH + TX + IDEMPOTENCIA + MENOS AWAITS
================================================================================
Problema: ~11 awaits seriales, sin tx, getActiveCashSession x2, getOrder x2.
Objetivo: <=6 awaits principales, todo en db.transaction, idempotencia sólida, finalization idempotente.

Idempotencia (OBLIGATORIA, sin migración si se puede):
- El request body debe aceptar idempotencyKey (string). Frontend lo genera al abrir el modal “Pagar”.
- Guardar el idempotencyKey en un campo EXISTENTE de payments de forma consistente:
  - Preferir payments.metadata JSON si existe => metadata.idempotencyKey
  - Si no: usar payments.notes/reference/description (el campo real) con formato EXACTO: "IDEMP:<key>"
- Check idempotencia: antes de insertar pago, buscar payment existente para (order_id=X y key match y status válido).
- Importante: si existe payment por key pero la order NO quedó finalizada (status no PAID), ejecutar finalization idempotente dentro de tx y devolver (no duplicar).

Implementación:
- const now = new Date() (una sola vez).
- Cachear getActiveCashSession: 1 vez.
- Cachear getOrder: 1 vez.
- Dentro de db.transaction(async (tx)=>{...}):
  1) Validaciones con lecturas no duplicadas.
  2) Idempotency check:
     - si payment existe por key => ejecutar finalizePaymentTx(...) (idempotente) y return.
  3) Pago total:
     - Batch update order_items por order_id SOLO en pago total:
       UPDATE order_items SET status='PAID' WHERE order_id=X AND status NOT IN ('VOIDED') (ajusta enums reales)
     - Batch update ledger por order_id:
       UPDATE sales_ledger_items SET status='PAID', paid_at=now WHERE order_id=X AND status != 'VOIDED'
  4) Split cleanup (si aplica):
     - 1 query splitAccountIds por order_id=X
     - DELETE split_items WHERE split_id IN (...)
     - DELETE split_accounts WHERE order_id=X
  5) Service charge: construir array y 1 insert batch (sin loop await).
  6) Insert payment (1) guardando idempotencyKey.
  7) Insert audit events en batch si aplica.
  8) finalizePaymentTx(...) para cerrar orden/timestamps/balances.
- Reducir awaits redundantes (no llamar getOrder/getActiveCashSession dos veces).

Acceptance:
- <=6 awaits principales; tx; idempotencia evita doble cobro; finalization idempotente; p95 <200ms.
