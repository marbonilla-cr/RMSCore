# Prompt — Integración QuickBooks Online (QBO)

## Contexto del sistema

Stack: Express.js + TypeScript, Drizzle ORM, PostgreSQL, React + Vite, Tailwind CSS, shadcn/ui, wouter.
Timezone: `America/Costa_Rica` (UTC-6). Todos los pagos están en la tabla `payments`. Los pagos del RMS tienen `origin = 'SYSTEM'`. Los históricos de Loyverse tienen `origin = 'LOYVERSE'` — **estos NO se sincronizan con QBO** (ya están en QBO vía Amaka).

**Principio guía:** Si QBO falla, el cobro no falla. La sincronización es asíncrona y con reintentos.

-----

## Alcance

1. Base de datos — 3 tablas nuevas
1. Servicio QBO — cliente OAuth + API calls

## 1. BASE DE DATOS

### Tabla: `qbo_config` — credenciales y estado de conexión (una sola fila)

```typescript
{
  id:                serial primary key,
  // OAuth tokens — guardar encriptados con AES-256 usando env var QBO_ENCRYPT_KEY
  accessToken:       text,           // encriptado
  refreshToken:      text,           // encriptado
  realmId:           varchar(100),   // ID empresa en QBO (obtenido en OAuth)
  tokenExpiresAt:    timestamp,      // accessToken expira en 1h
  // Config contable — un QBO account ID por método de pago
  depositAccountCash:   varchar(50), // cuenta destino para pagos en efectivo
  depositAccountCard:   varchar(50), // cuenta destino para pagos con tarjeta
  depositAccountSinpe:  varchar(50), // cuenta destino para pagos SINPE
  // Config impuesto global
  taxCodeRef:        varchar(50),    // código de impuesto en QBO (ej: "TAX")
  // Estado
  isConnected:       boolean default false,
  connectedAt:       timestamp,
  lastTokenRefresh:  timestamp,
  // Fecha de corte — órdenes ANTES de esta fecha NO se sincronizan (ya están en QBO vía Amaka)
  syncFromDate:      date,           // se configura manualmente una sola vez
}
```

### Tabla: `qbo_category_mapping` — mapeo categoría RMS → item QBO

```typescript
{
  id:            serial primary key,
  categoryId:    integer not null references categories(id),
  qboItemId:     varchar(50) not null,    // ID del item en QBO
  qboItemName:   varchar(200),            // nombre para mostrar en UI
  updatedAt:     timestamp default now(),
  // unique(categoryId)
}
```

### Tabla: `qbo_sync_log` — registro de cada intento de sync por orden

```typescript
{
  id:              serial primary key,
  paymentId:       integer not null references payments(id),
  orderId:         integer not null,
  status:          varchar(20) not null default 'PENDING',
  // PENDING | SUCCESS | FAILED | VOIDED
  qboReceiptId:    varchar(100),   // SalesReceipt ID devuelto por QBO (cuando SUCCESS)
  qboReceiptNumber: varchar(50),   // número de documento en QBO
  errorMessage:    text,
  attempts:        integer default 0,
  createdAt:       timestamp default now(),
  syncedAt:        timestamp,
  nextRetryAt:     timestamp,
  // unique(paymentId) — un pago = un registro de sync
}
```

## 3. SERVICIO QBO (`server/services/quickbooks.ts`)

Usar la librería oficial: `npm install node-quickbooks intuit-oauth`

### 3.1 OAuth

```typescript
// Generar URL de autorización
function getAuthUrl(): string

// Intercambiar code por tokens (callback de OAuth)
async function handleOAuthCallback(code: string, realmId: string): Promise<void>
// Guarda accessToken + refreshToken encriptados en qbo_config

// Refrescar accessToken si está por vencer (< 5 min para expirar)
async function ensureFreshToken(): Promise<string>
// Llama QBO token refresh endpoint, actualiza qbo_config

// Encriptar / desencriptar tokens
function encrypt(text: string): string
function decrypt(text: string): string
// Usar crypto.createCipheriv con AES-256-CBC y QBO_ENCRYPT_KEY
```

### 3.2 Obtener listas de QBO (para poblar dropdowns en UI)

```typescript
// Traer todos los items (productos/servicios) de QBO
async function getQBOItems(): Promise<{ id: string; name: string; type: string }[]>

// Traer todas las cuentas bancarias/efectivo de QBO
async function getQBOAccounts(): Promise<{ id: string; name: string; accountType: string }[]>

// Traer todos los payment methods de QBO
async function getQBOPaymentMethods(): Promise<{ id: string; name: string }[]>

// Traer tax codes de QBO
async function getQBOTaxCodes(): Promise<{ id: string; name: string; rate: number }[]>
```

### 3.3 Crear SalesReceipt

```typescript
async function createSalesReceipt(orderId: number, paymentId: number): Promise<string>
// Devuelve el QBO SalesReceipt ID

// Lógica interna:
// 1. Obtener orden completa: items + taxes + discounts + payment method
// 2. Agrupar items por categoría
// 3. Para cada grupo → una línea con:
//    - ItemRef: qboItemId del mapeo de esa categoría
//    - Amount: suma de (precio * qty) de todos los items del grupo
//    - Qty: 1 (la línea representa el total de la categoría)
//    - UnitPrice: el Amount
//    - TaxCodeRef: taxCodeRef de qbo_config
// 4. Si un item no tiene categoría mapeada → usar item "Sin categoría" si existe mapeo,
//    si no → loguear warning y omitir (no fallar el receipt)
// 5. Calcular descuentos: si hay descuentos en la orden → una línea negativa adicional
// 6. DepositToAccountRef: según payment method (CASH/CARD/SINPE → cuenta configurada)
// 7. TxnDate: business_date de la orden (fecha local CR, no UTC)
// 8. DocNumber: ticketNumber de la orden (ej: "0234")
// 9. CustomerRef: cliente genérico "Cliente Mostrador" — crear en QBO si no existe
// 10. POST a /v3/company/{realmId}/salesreceipt
// 11. Guardar qboReceiptId en qbo_sync_log con status SUCCESS
// 12. Si falla → actualizar qbo_sync_log con status FAILED + errorMessage + nextRetryAt
```

### 3.4 Void SalesReceipt (cuando se anula un pago en el RMS)

```typescript
async function voidSalesReceipt(paymentId: number): Promise<void>
// Busca el qboReceiptId en qbo_sync_log para ese paymentId
// Si existe y status = SUCCESS → POST /v3/company/{realmId}/salesreceipt?operation=void
// Actualiza qbo_sync_log status = VOIDED
// Si no existe o ya es VOIDED → no hacer nada (idempotente)
```

### 3.5 Cola de reintentos

```typescript
async function retryPendingSync(): Promise<void>
// Busca en qbo_sync_log donde:
//   status = 'PENDING' OR (status = 'FAILED' AND nextRetryAt < now() AND attempts < 5)
// Para cada uno → llamar createSalesReceipt(orderId, paymentId)
// Límite: procesar máximo 20 por corrida para no saturar QBO
// nextRetryAt con backoff exponencial: 5min, 15min, 1h, 4h, 24h
```

Registrar `retryPendingSync` como job con `setInterval` cada 5 minutos en el arranque del servidor:

```typescript
// En server/index.ts al iniciar:
setInterval(() => retryPendingSync().catch(console.error), 5 * 60 * 1000)
```

## 4. RUTAS BACKEND (`server/routes/quickbooks.ts`)

Todas requieren autenticación excepto el callback de OAuth.

```
GET    /api/qbo/status
       Respuesta: { isConnected, realmId, connectedAt, syncFromDate,
                    hasCategoryMappings, hasDepositAccounts, hasTaxCode }

GET    /api/qbo/auth-url
       Respuesta: { url } — URL de autorización de Intuit

GET    /api/qbo/callback
       Query: code, realmId, state
       Llama handleOAuthCallback(), redirige a /admin/quickbooks?connected=true
       NO requiere autenticación (es callback de Intuit)

DELETE /api/qbo/disconnect
       Borra tokens de qbo_config, isConnected = false

GET    /api/qbo/items          → getQBOItems()
GET    /api/qbo/accounts       → getQBOAccounts()
GET    /api/qbo/tax-codes      → getQBOTaxCodes()

GET    /api/qbo/mappings
       Respuesta: categorías del RMS con su mapeo QBO actual

PUT    /api/qbo/mappings
       Body: [{ categoryId, qboItemId, qboItemName }]
       Upsert masivo de mappings

PUT    /api/qbo/config
       Body: { depositAccountCash, depositAccountCard, depositAccountSinpe,
               taxCodeRef, syncFromDate }
       Actualiza qbo_config

GET    /api/qbo/sync-log
       Query: status, limit (default 50), offset
       Respuesta: registros de qbo_sync_log con info de orden

POST   /api/qbo/retry-failed
       Dispara retryPendingSync() manualmente
       Respuesta: { queued: number }

POST   /api/qbo/initial-sync
       Body: { fromDate: 'YYYY-MM-DD' }
       Lógica:
         1. Validar que QBO esté conectado y mappings configurados
         2. Buscar todos los payments donde:
            origin = 'SYSTEM'
            AND status = 'PAID'
            AND business_date >= fromDate
            AND paymentId NOT IN (SELECT paymentId FROM qbo_sync_log WHERE status = 'SUCCESS')
         3. Insertar todos en qbo_sync_log con status = 'PENDING'
         4. Devolver { queued: number } inmediatamente
         5. El job de reintentos los procesa en background (cada 5 min)
         // NO procesar síncronamente — devolver rápido y dejar al job
```

## 5. HOOK EN POS — Disparar sync al cobrar

Modificar `server/routes/pos.ts` en los endpoints de pago.

**Después de cada pago exitoso**, llamar de forma asíncrona (fire and forget):

```typescript
// En POST /api/pos/pay — después de confirmar el pago:
if (qboConfig?.isConnected && payment.businessDate >= qboConfig.syncFromDate) {
  // Insertar en sync log como PENDING
  await db.insert(qboSyncLog).values({
    paymentId: payment.id,
    orderId: order.id,
    status: 'PENDING',
  }).onConflictDoNothing() // idempotente

  // Intentar sync inmediato en background (no await)
  createSalesReceipt(order.id, payment.id)
    .catch(err => console.error('QBO sync failed, will retry:', err.message))
}

// Lo mismo para POST /api/pos/pay-split
```

**Cuando se anula un pago** (`POST /api/pos/void-payment/:id`):

```typescript
// Después de confirmar el void:
voidSalesReceipt(paymentId)
  .catch(err => console.error('QBO void failed:', err.message))
```

**Importante:** estos calls son fire-and-forget. El endpoint de cobro responde al cajero sin esperar a QBO.

## 8. REGLAS DE NEGOCIO CRÍTICAS

### Solo sincronizar pagos del RMS

```typescript
// SIEMPRE verificar antes de sync:
payment.origin === 'SYSTEM'
AND payment.businessDate >= qboConfig.syncFromDate
// Nunca sincronizar origin = 'LOYVERSE' — ya están en QBO vía Amaka
```

### Idempotencia

- `qbo_sync_log` tiene unique en `paymentId` → `onConflictDoNothing()` en todos los inserts
- Antes de crear un receipt, verificar que no existe ya un `SUCCESS` para ese `paymentId`
- Si ya existe un `SUCCESS` → no crear duplicado, retornar el `qboReceiptId` existente

### El cobro nunca falla por culpa de QBO

- Todo sync es fire-and-forget o en background
- Si QBO está caído → el pago se registra en el RMS normal, `qbo_sync_log` queda `PENDING`, el job lo reintenta

### TxnDate = fecha de negocio CR

```typescript
// Usar business_date de la orden (date local CR), NO el timestamp UTC
TxnDate: order.businessDate  // "2026-02-20", ya es fecha local CR
```

### Tokens encriptados

- Nunca guardar `accessToken` o `refreshToken` en texto plano en la DB
- Usar `AES-256-CBC` con `QBO_ENCRYPT_KEY` del env
- Si `QBO_ENCRYPT_KEY` no está configurado → lanzar error al iniciar servidor

### Renovación de tokens

- QBO `accessToken` expira en 1 hora
- `refreshToken` expira en 101 días
- `ensureFreshToken()` debe llamarse antes de CADA llamada a QBO API
- Si el refresh falla (token expirado) → marcar `isConnected = false` en `qbo_config` + loguear error
