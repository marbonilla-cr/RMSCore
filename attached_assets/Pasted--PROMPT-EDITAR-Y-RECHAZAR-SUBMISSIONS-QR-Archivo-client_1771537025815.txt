## CONTEXTO

Cuando un cliente ordena por QR, el mesero ve la submission en la vista
de orden con un botón "Enviar a cocina". El problema es que no hay forma
de corregir errores antes de enviar.

Se necesitan **dos acciones nuevas** además del botón existente:
- **Editar** — ajustar cantidades o eliminar ítems del submission antes
  de aceptarlo
- **Rechazar** — descartar el submission sin enviarlo a cocina

## PARTE 2 — MUTATIONS NUEVAS (agregar junto a las mutations existentes)

### Rechazar submission
```tsx
const rejectSubmissionMutation = useMutation({
  mutationFn: async (submissionId: number) => {
    return apiRequest("DELETE", `/api/waiter/qr-submissions/${submissionId}`);
  },
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ["/api/tables", tableId, "current"] });
    queryClient.invalidateQueries({ queryKey: ["/api/waiter/tables"] });
    toast({ title: "Pedido QR rechazado" });
  },
  onError: (err: any) => {
    toast({ title: "Error", description: err.message, variant: "destructive" });
  },
});
```

### Aceptar submission con payload editado
La función `acceptSubmissionMutation` existente llama a `accept-v2` sin body.
Agregar una mutation separada que envíe el payload modificado:

```tsx
const acceptEditedSubmissionMutation = useMutation({
  mutationFn: async ({ submissionId, items }: { submissionId: number; items: any[] }) => {
    const res = await apiRequest(
      "POST",
      `/api/waiter/qr-submissions/${submissionId}/accept-v2`,
      { editedItems: items }   // El backend puede ignorarlo si no lo implementa aún,
    );                          // pero el frontend lo envía para cuando esté listo.
    return res.json();
  },
  onSuccess: (data: any) => {
    setEditingSubmissionId(null);
    setEditedPayloadItems([]);
    setExpandedSubmissionId(null);
    if (data.activeOrder) {
      queryClient.setQueryData(["/api/tables", tableId, "current"], {
        table: data.table || currentView?.table,
        activeOrder: data.activeOrder,
        orderItems: data.orderItems || [],
        pendingQrSubmissions: data.pendingQrSubmissions || [],
      });
    } else {
      queryClient.invalidateQueries({ queryKey: ["/api/tables", tableId, "current"] });
    }
    queryClient.invalidateQueries({ queryKey: ["/api/waiter/tables"] });
    toast({ title: "Pedido QR aceptado y enviado a cocina" });
  },
  onError: (err: any) => {
    toast({ title: "Error", description: err.message, variant: "destructive" });
  },
});
```

## PARTE 5 — BACKEND (instrucción lógica para el agente)

El frontend hace dos llamadas que el backend debe soportar:

### DELETE `/api/waiter/qr-submissions/:id`
- Buscar el submission por ID
- Verificar que su estado sea PENDING (no se puede rechazar uno ya aceptado)
- Eliminar o marcar como REJECTED el registro
- Retornar 200 OK
- Invalidar el conteo de submissions pendientes de la mesa

### POST `/api/waiter/qr-submissions/:id/accept-v2` con body `{ editedItems }`
- El endpoint `accept-v2` ya existe y funciona
- Si llega `editedItems` en el body (array no vacío), usar esos ítems
  en lugar del `payloadSnapshot` original al crear los order items
- Si no llega `editedItems` o llega vacío, comportarse exactamente igual
  que antes (compatibilidad hacia atrás garantizada)
- Esto permite que el frontend editado funcione sin romper el flujo normal

## COMPORTAMIENTO FINAL

**Vista normal (expandida):**
```
[🗑 Rechazar]  [⚙ Editar]  [→ Enviar a cocina ──────]
```

**Vista en edición:**
- Badge "Editando" en ámbar en el header
- Header no colapsa mientras se edita
- Cada ítem muestra: [−] qty [+] nombre [✕]
- Si qty baja a 0 el ítem desaparece de la lista
- Si se eliminan todos los ítems: mensaje en rojo + botón "Enviar" deshabilitado
- Botones: [✕ Cancelar]  [→ Enviar a cocina]

**Rechazo:**
- Botón 🗑 dispara directamente sin confirmación adicional
  (el submission es pequeño, el error es reversible volviendo a pedir)
