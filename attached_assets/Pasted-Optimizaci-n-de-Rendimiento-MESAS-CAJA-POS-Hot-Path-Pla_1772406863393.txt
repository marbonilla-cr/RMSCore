T003: POST /api/pos/pay — batch + tx + idempotencia + menos awaits
========================================================
Problema: ~11 awaits seriales, sin tx, getActiveCashSession x2, getOrder x2.
Objetivo: <=6 awaits principales, db.transaction, idempotencia.

Idempotencia mínima:
- Aceptar un idempotencyKey en el body (string). El frontend lo genera al abrir el modal de pagar.
- Si NO quieres migración: usa un campo existente en payments (ej. reference/notes/meta JSON) para guardarlo.
- Antes de crear payment: buscar si ya existe un payment con ese idempotencyKey para ese orderId y status=PAID/COMPLETED (según tu modelo). Si existe, devolver ese resultado y NO duplicar.

Implementación (server/routes.ts):
- const now = new Date() una vez.
- Cachear getActiveCashSession: 1 vez.
- Cachear getOrder: 1 vez.
- Dentro de db.transaction(async (tx)=>{...}):
  1) Si order ya está PAID/closed, devolver early (sin insertar).
  2) Idempotency check (si key viene):
     - SELECT payments WHERE order_id=X AND (idempotencyKey match en campo disponible) LIMIT 1
     - Si existe: return same response.
  3) Batch updates:
     - UPDATE order_items SET status='PAID' WHERE order_id=X AND status NOT IN ('VOIDED') (ajusta status real)
     - UPDATE sales_ledger_items SET status='PAID', paid_at=now WHERE order_id=X AND status != 'VOIDED'
  4) Batch delete splits:
     - 1 query: split account ids por order_id=X
     - DELETE split_items WHERE split_id IN (...)
     - DELETE split_accounts WHERE order_id=X
  5) Service charge: 1 insert batch.
  6) Payment insert: 1 insert payments (incluye idempotencyKey guardado).
  7) Audit insert: 1 insert batch si aplica.
  8) Update order status/cierre: 1 update (status=PAID, paid_at=now, updated_at=now, etc.)
- Eliminar awaits redundantes.

Acceptance: <=6 awaits principales; tx; idempotencia evita doble cobro; p95 <200ms.
