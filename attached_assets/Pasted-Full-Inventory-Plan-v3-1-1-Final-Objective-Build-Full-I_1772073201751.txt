Full Inventory — Plan v3.1.1 (Final)
Objective
Build Full Inventory module with AP→EP conversions, production batches, recipes referencing EP/AP, auto-deduction on SENT (FULL-first then BASIC), physical counts, and purchase orders. All with DB transactions, row locks, and idempotency.

Key Design Decisions
No separate inv_ep_items table — reuse inv_items with itemType column ('AP'|'EP')
Single movements table — extend existing inv_movements with itemType, standardized reference fields (refType, refId), and canonical movementType values
Canonical movementType values (text enum used across the module):
CONSUME_AP — AP stock consumed (production batch or direct recipe consumption)
CONSUME_EP — EP stock consumed (recipe deduction on SENT)
PRODUCE_EP — EP stock produced (production batch output)
ADJUST_AP — Manual AP stock adjustment (physical count or manual)
ADJUST_EP — Manual EP stock adjustment (physical count or manual)
RECEIVE_AP — AP stock received (purchase order receipt)
REVERSE_CONSUME_AP — Reversal of CONSUME_AP (void/cancel)
REVERSE_CONSUME_EP — Reversal of CONSUME_EP (void/cancel)
(Future, not implemented in this plan): TRANSFER_AP, TRANSFER_EP — inter-location transfers
FULL-then-BASIC deduction — on SENT: run FULL recipe deduction first, only if OK then run BASIC (decrementPortions). If FULL fails, block SENT entirely.
Transactions + FOR UPDATE locks — production batches and deduction use DB transactions with row-level locks, lock ordering by (itemType, invItemId) to avoid deadlocks, ensure stock rows exist with INSERT ON CONFLICT DO NOTHING before SELECT FOR UPDATE
Idempotency — inventory_deductions table with unique orderItemId, basicDeductedAt column, productId + orderItemQty snapshot for deterministic reversals. Items without recipes still create a deduction row (empty payload) to protect BASIC idempotency.
Backward compatible — products without recipes just run BASIC deduction as before
Existing Infrastructure (reuse)

T005: Auto-deduction on SENT + idempotency + reversal
Blocked By: [T003, T004]
Details:
Create server/inventory-deduction.ts with two main functions:

onOrderItemsConfirmedSent(orderId, orderItemIds[], userId): For each orderItemId, IN DB TRANSACTION:

a) Idempotency gate (SELECT inventory_deductions WHERE orderItemId=? FOR UPDATE):

If exists AND status='CONSUMED' AND basicDeductedAt IS NOT NULL → COMMIT, skip (fully done)
If exists AND status='CONSUMED' AND basicDeductedAt IS NULL → skip FULL, run BASIC only, set basicDeductedAt, COMMIT (crash recovery)
If exists AND status='REVERSED' → skip. Policy: orderItemId must NEVER be reused for a re-sent item. Any re-send must create a new order item row (new orderItemId).
If not exists → proceed to step b
b) Snapshot: SELECT order_items row FOR UPDATE to lock it, then read productId + qty (orderItemQty). This lock happens before recipe resolution and before stock locks to ensure consistent snapshot.

c) Resolve recipe:

Find active recipe for the menu product
If no recipe exists → create inventory_deductions row with empty consumptionPayload=[], then run BASIC and set basicDeductedAt. COMMIT. (No-recipe items still idempotent for BASIC)
If recipe exists → proceed to step d
d) Build consumption list from recipe lines:

For each line: consumption = orderItemQty * qtyBasePerMenuUnit * (1 + wastePct/100)
Aggregate by (itemType, invItemId) — combine duplicate items
e) Lock ordering to avoid deadlocks:

Sort aggregated consumption keys by (itemType ASC, invItemId ASC)
For each key:
Ensure stock row exists: INSERT INTO inv_stock_ap/ep(...) ON CONFLICT DO NOTHING
SELECT qtyOnHand FROM inv_stock_ap/ep WHERE organizationId=1 AND locationId=1 AND invItemId=? FOR UPDATE
Validate sufficient qty (qtyOnHand >= consumption). If insufficient → throw error, block SENT, do NOT run BASIC
f) Apply stock updates in same locked order:

Decrement stock (EP or AP based on line itemType)
Insert movements (CONSUME_EP or CONSUME_AP, refType='ORDER_ITEM', refId=orderItemId)
g) Insert inventory_deductions row:

consumptionPayload = array of {invItemId, itemType, qty} (qty as numeric-as-string)
productId = snapshot productId
orderItemQty = snapshot qty
status = 'CONSUMED'
h) Run BASIC deduction:

decrementPortions(productId, orderItemQty, orderItemId, userId)
Set basicDeductedAt = now() on the deduction row
COMMIT
i) If FULL fails (insufficient stock):

Throw error with message listing insufficient items
Do NOT create deduction row, do NOT run BASIC, do NOT allow SENT
onOrderItemsVoided(orderItemIds[], userId): For each orderItemId, IN DB TRANSACTION:

SELECT inventory_deductions WHERE orderItemId=? FOR UPDATE
If not found OR status='REVERSED' → skip (idempotent)
If status='CONSUMED': a. Restore FULL: iterate consumptionPayload, lock stock rows in sorted order (itemType ASC, invItemId ASC) using WHERE organizationId=1 AND locationId=1 AND invItemId=?, increment stock rows, insert REVERSE_CONSUME_EP/REVERSE_CONSUME_AP movements (refType='ORDER_ITEM', refId=orderItemId) b. Set status='REVERSED', reversedAt=now() c. If basicDeductedAt IS NOT NULL:
incrementPortions(productId from row, orderItemQty from row, orderItemId, userId) — BASIC reversal using snapshot values d. COMMIT
Wire into SENT triggers (server/routes.ts + qr-subaccount-routes.ts):

POS add-items (~line 1354): inside if (sendToKds), call onOrderItemsConfirmedSent BEFORE creating kitchen ticket. If throws → return 400 error, don't create ticket, don't mark SENT
Waiter send-round (~line 2000): after creating items as SENT, call deduction. If throws → rollback items to previous state
QR accept (~line 2095): after marking SENT, call deduction
QR subaccount accept (qr-subaccount-routes.ts ~line 457): same
Wire into void triggers:

Waiter void (~line 2213): call onOrderItemsVoided([itemId]) — replaces standalone incrementPortions call
Hard delete (~line 2269): call onOrderItemsVoided([itemId]) before delete
Order void (~line 3954): call onOrderItemsVoided(allItemIds) — replaces per-item incrementPortions
Files: server/inventory-deduction.ts, server/routes.ts, server/qr-subaccount-routes.ts

Acceptance:

Critical Rules
No inv_ep_items table — EP items live in inv_items with itemType='EP'
Single movements table (inv_movements) with canonical movementType values (CONSUME_AP, CONSUME_EP, PRODUCE_EP, ADJUST_AP, ADJUST_EP, RECEIVE_AP, REVERSE_CONSUME_AP, REVERSE_CONSUME_EP) and standardized refType/refId fields
FULL deduction runs first; BASIC only after FULL succeeds
All stock mutations in DB transactions with FOR UPDATE locks, lock ordering by (itemType, invItemId)
Ensure stock rows with INSERT ON CONFLICT DO NOTHING before SELECT FOR UPDATE
All new tables have organizationId/locationId DEFAULT 1
No WAC/FIFO — simple last-cost pricing
Backward compatible: products without recipes create deduction row with empty payload, then run BASIC only
Reversals use snapshot (productId + orderItemQty) from inventory_deductions row
Movements reference fields: refType (BATCH, ORDER_ITEM, PO_RECEIPT, PHYSICAL_COUNT, MANUAL) + refId
orderItemId reuse policy: a REVERSED orderItemId must NEVER be reused for a re-sent item. Any re-send must create a new order item row with a new orderItemId.
All stock queries must use the full UNIQUE key: WHERE organizationId=1 AND locationId=1 AND invItemId=?
