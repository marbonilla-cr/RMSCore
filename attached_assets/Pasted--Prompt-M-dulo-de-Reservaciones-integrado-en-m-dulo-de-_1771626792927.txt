# Prompt — Módulo de Reservaciones (integrado en módulo de Mesas existente)

## Principio guía
**No crear módulos nuevos.** Todo se integra en las pantallas que el salonero ya usa a diario. El cambio debe sentirse como una extensión natural de `tables.tsx`, no como una funcionalidad separada.

---

## Contexto del sistema
Stack: React + Vite, Tailwind CSS, shadcn/ui, wouter, Express.js + TypeScript, Drizzle ORM, PostgreSQL, WebSocket (`ws`). Timezone: `America/Costa_Rica` (UTC-6). Timestamps: UTC `timestamp without time zone`. Patrones: TanStack Query v5, `apiRequest()`, `wsManager`, polling 5s.

```typescript
{

  reservationCode:     varchar(20) not null unique,  // "RES-2026-0043"

  cancelledAt:         timestamp,
  cancellationReason:  text,
  createdAt:           timestamp default now(),
  updatedAt:           timestamp default now(),

  // Notificaciones

  reminderSentAt:      timestamp,
}
```

## 2. BACKEND

### 2.1 Modificar `GET /api/waiter/tables`

Agregar en la respuesta de cada mesa el campo `upcomingReservation`:

```typescript
interface TableView {
  // ... todos los campos existentes sin cambios ...

  // NUEVO:
  upcomingReservation: {
    id:           number
    guestName:    string
    partySize:    number
    reservedDate: string   // "2026-02-20"
    reservedTime: string   // "19:30"
    status:       string   // PENDING | CONFIRMED
    minutesUntil: number   // minutos desde ahora hasta reservedDate+reservedTime
  } | null
  // Solo incluir la reserva más próxima con status PENDING o CONFIRMED
  // Solo si está dentro de las próximas 24 horas
}
```

**No romper ningún campo existente.**

### 2.2 Nuevas rutas: `/api/reservations`

Todas requieren autenticación (sesión activa).

```
GET    /api/reservations
       Query params:
         date     (YYYY-MM-DD, required)
         tableId  (number, optional — filtrar por mesa)
         status   (string, optional — filtrar por status)
       Respuesta: Reservation[] con campo table: { id, tableName, tableCode }

GET    /api/reservations/:id

POST   /api/reservations
       Body: { guestName, guestPhone, guestEmail?, partySize,
               reservedDate, reservedTime, tableId?, notes?, durationMinutes? }
       - Genera reservationCode automático: "RES-YYYY-NNNN"
       - Calcula durationMinutes desde reservation_duration_config si no se envía
       - Valida conflicto: no puede haber otra reserva CONFIRMED/SEATED en la
         misma mesa cuyo intervalo [reservedTime, reservedTime+duration) se solape
       - status inicial: CONFIRMED (salonero la crea ya confirmada)
       - Emite WS: reservation_updated

PATCH  /api/reservations/:id
       Body parcial: { guestName?, guestPhone?, guestEmail?, partySize?,
                       reservedDate?, reservedTime?, tableId?, notes?,
                       durationMinutes? }
       - Solo si status es PENDING o CONFIRMED
       - Revalida conflictos si cambia mesa, fecha u hora
       - Emite WS: reservation_updated

PATCH  /api/reservations/:id/status
       Body: { status: string, reason?: string }
       Transiciones válidas:
         PENDING    → CONFIRMED | CANCELLED
         CONFIRMED  → SEATED | NO_SHOW | CANCELLED
         SEATED     → (terminal, no puede cambiar)
         WAITING    → CONFIRMED | CANCELLED
       - Registra seatedAt si status → SEATED
       - Registra cancelledAt + cancellationReason si → CANCELLED
       - Emite WS: reservation_updated

GET    /api/reservations/availability
       Query: date (YYYY-MM-DD), partySize, tableId? (optional)
       Respuesta: lista de mesas disponibles para esa fecha/partySize,
                  considerando conflictos de reservas existentes y capacidad
       Una mesa está disponible si:
         capacity >= partySize
         AND no tiene reserva CONFIRMED/SEATED en ese día sin tableId especificado,
             o sin conflicto de horario si se pide disponibilidad por hora
```

### 2.3 Ruta pública (sin autenticación)

```
POST   /api/public/reservations
       Body: { guestName, guestPhone, guestEmail?, partySize,
               reservedDate, reservedTime, tableId?, notes? }
       - status inicial: PENDING
       - Valida conflictos igual que la ruta autenticada
       - Rate limiting: 5 req/hour por IP
       - Envía email de confirmación si guestEmail está presente
         (usar Nodemailer ya configurado en el sistema)
       - Responde: { reservationCode, message }

GET    /api/public/reservations/available-times
       Query: date (YYYY-MM-DD), partySize
       Respuesta: slots de tiempo disponibles (cada 30min entre 11:00 y 22:00)
                  con mesas sugeridas para cada slot
```

### 2.4 WebSocket — nuevo evento

Emitir desde el backend cuando cambia cualquier reserva:
```typescript
{ type: "reservation_updated", data: { reservationId, tableId, status } }
```

```
client/src/components/reservations/

  ReservationFormDialog.tsx  ← dialog de crear/editar reserva
```

1. Un campo `upcomingReservation` en el tipo `TableView`
2. Badge en la tarjeta de mesa
3. Un botón en el header que abre `ReservationsSheet`
4. Los dos imports

```typescript

  if (!open) return
  const unsub = wsManager.on("reservation_updated", () => {
```

**Header del sheet:**

**Sección "En mesa"** — status SEATED

### 3.3 `ReservationFormDialog.tsx`


Dialog de shadcn/ui, ancho máximo 480px.

**Campos:**
- Nombre del cliente (required)
- Teléfono / WhatsApp (required)
- Email (opcional)
- Cantidad de personas — input numérico min 1
- Fecha — `<input type="date">` hasta 30 días adelante

- Nueva: `POST /api/reservations` → status inicial CONFIRMED
- Edición: `PATCH /api/reservations/:id`
- On success: `queryClient.invalidateQueries(["/api/reservations"])` + `queryClient.invalidateQueries(["/api/waiter/tables"])` + cerrar dialog

## 4. PÁGINA PÚBLICA — `/reserve`

Ruta pública en `App.tsx`, igual que `/qr/:tableCode`.

Flujo simple en 4 pasos con los mismos tokens visuales del sistema QR:
```
bg: #faf8f4  card: #fff  border: #e8e2d9
text: #1a1612  text2: #5a524a  text3: #9a8f83
acc: #2d6a4f  accD: #e8f5ee
```

### S0 — Fecha, hora y personas
- Selector de fecha (calendar simple, hasta 30 días)
- Selector de personas: botones − / número / +
- Selector de hora: grid de pills con los slots disponibles (fetch `/api/public/reservations/available-times`)
  - Pills verdes = disponibles, pills grises = sin disponibilidad
- Botón "Continuar →"

### S1 — Mesa (opcional)
- Select de mesas disponibles para esa fecha/hora/partySize
- Mostrar nombre de mesa + capacidad
- Opción "Sin preferencia (el restaurante asigna)"
- Botón "Continuar →"

### S2 — Datos del cliente
- Nombre completo (required)
- Teléfono / WhatsApp (required)
- Email (opcional)
- Notas especiales (opcional) — placeholder: "Cumpleaños, alergias, preferencia de ubicación..."
- Botón "Confirmar reserva" → `POST /api/public/reservations`

### S3 — Confirmación enviada
- Ícono ✓ animado
- "¡Reserva recibida!"
- Código prominente: `RES-2026-0043`
- "El restaurante confirmará tu reserva pronto."
- Si se proporcionó email: "Te enviamos los detalles a [email]"
- Botón "Hacer otra reserva" → vuelve a S0
